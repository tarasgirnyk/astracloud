# Звіт: механізм підключення CSS у темі astra (file_css)

Дата: 2026-07-17. Діагностика read-only; нічого не змінювалось.

## TL;DR

1. `file_css` живе **в БД HostBill** (механізм themeconfig), у файлах теми його значення немає; PHP-логіка (плагін `function.themeconfig.php` і адмінка) закодована ionCube — валідацію бачимо лише емпірично.
2. `?v=` — це **глобальний токен `{$hb_version}`**, спільний для CSS і JS; він НЕ є хешем вмісту файлу і не привʼязаний до mtime. Покладатись на нього для версіонування власного файлу не можна.
3. ⚠️ **Виявлена жива проблема поза межами завдання:** тема astra **вже активована** в HostBill (custom.min.css скомпільовано сьогодні 16:11, адмінка `theme_config id=2`), але **зовнішній сайт її не віддає**: `cp.astra.in.ua` (45.87.90.131) — це окремий фронт-вузол, на якому папки `templates/astra/` немає. Всі astra-статики зовні дають 404, і HostBill на фронті **відкотився на тему 2019**. Локально (цей сервер, 172.17.255.6) astra рендериться і віддається коректно. Без синхронізації файлів на фронт будь-який власний CSS не запрацює.

## 1. Де живе file_css

* У файлах теми astra `file_css` згадується рівно один раз — у рядку підключення ([header.shared.tpl:59](../home/hostbill/public_html/templates/astra/header.shared.tpl)):
  ```smarty
  <link rel="stylesheet" type="text/css"
    href="{themeconfig prefix=$template_dir section=css variable=file_css default="dist/css/app.min.css"}?v={$hb_version}">
  ```
* `configuration/configuration.json` теми astra **байт-у-байт ідентичний** конфігу 2019 (навіть `"name": "2019 theme"` не перейменовано). Секція `css` описує лише option `custom_css`; **option `file_css` у конфігу відсутній** — тобто це не «настройка з форми», а службова змінна, яку адмінка виставляє програмно після компіляції кастомізації.
* Отже: **значення file_css зберігається в БД** (сховище themeconfig, per-theme). У БД не лазив (за умовою); підтверджено непрямо: у файлах значення немає, а живий рендер віддає `custom.min.css` замість дефолтного `app.min.css`.
* Реалізація `{themeconfig}`: `includes/libs/smarty/plugins/function.themeconfig.php` — **ionCube-закодований** (як і всі PHP-плагіни теми в `templates/astra/plugins/`). Вихідної логіки не прочитати.

## 2. Поточний стан: що реально віддається

Живі перевірки (17 лип 2026, ~17:00 EEST):

| Запит | Результат |

| той самий запит **локально** (127.0.0.1, цей сервер) | CSS **`/templates/astra/dist/css/custom.min.css?v=1f52b425`** |

Пояснення розбіжності: `cp.astra.in.ua` → **45.87.90.131**, а цей сервер — **172.17.255.6** (за NAT/окремо). Фронт-вузол має власний nginx з таким самим конфігом (`location ^~ / { alias …; try_files $uri $uri/ @seohburl; }` + той самий paddle sub_filter) і **власну копію `public_html`, в якій немає `templates/astra/`** (клон робився сьогодні 12:48 лише тут). try_files на фронті не знаходить файл → запит падає в PHP → HostBill віддає 404-сторінку. А оскільки папки теми немає, HostBill на фронті **мовчки відкочується до теми 2019** — тому зовнішні відвідувачі досі бачать стару тему, хоча в БД активна astra.

Права ФС, SELinux (Disabled) і локальний nginx — не при чому: локально все віддається.

## 3. Як формується ?v=

* У шаблоні: `?v={$hb_version}` — **той самий токен** і для CSS (рядок 59), і для `dist/js/main.min.js` (рядок 86).
* Це **не** хеш вмісту файлу і **не** mtime: 8-hex значення однакове для різних файлів з різними датами; `includes/version` містить лише `version: 2024-02-14` (md5-кандидати від дат не збігаються — генерується всередині закодованого ядра).
* Емпірично: у попередній діагностиці було `b66a5e5a`, сьогодні — `1f52b425`, **однаковий на фронті й локально** (той самий БД-токен). Зміна збіглась із сьогоднішніми маніпуляціями з темою в адмінці (theme_config, компіляція custom.min.css о 16:11). Тобто токен оновлюється при діях адміністратора з темою/оновленнях HostBill — але **не при зміні файлу CSS на диску**.
* Наслідок для власної збірки: якщо просто перезаписувати свій CSS-файл, браузери зі старим `?v=` можуть тримати стару версію (nginx віддає ETag/Last-Modified, явних `Cache-Control` немає — ревалідація «мʼяка»; правило `expires 1y` у [main.conf:22](/etc/nginx/conf.d/main.conf) фактично мертве, бо `location ^~ /` перекриває regex-локацію). **Потрібен власний версіонований параметр** (хеш вмісту в query або в імені файлу).

## 4. Чи можна вказати довільний шлях у file_css

Що відомо напевно:

* Шаблон вставляє значення as-is: `href="{префікс template_dir}{значення file_css}?v=…"` — жодної валідації на рівні tpl.
* Видимої валідації в темі немає (плагіни ionCube), а в `configuration.json` поля file_css взагалі нема — отже, стандартного UI «вписати свій шлях» не існує; значення виставляє сама адмінка при збереженні кастомізації (ставить `dist/css/custom.min.css`).
* Технічно довільний відносний шлях усередині теми (наприклад `dist/css/astra.min.css`) працюватиме, якщо його записати в БД — але це запис у БД поза штатним UI, і **будь-яке наступне збереження Theme customization в адмінці перезапише значення назад** на `dist/css/custom.min.css`.

## 5. Канал custom_css

Фрагмент [header.shared.tpl:61-65](../home/hostbill/public_html/templates/astra/header.shared.tpl):

```smarty
{literal}
<style>
    {/literal}{themeconfig variable=custom_css default=""}{literal}
</style>
{/literal}
```

Поле «Custom CSS» з адмінки (секція `css` конфігурації, тип `css`) інлайниться **в `<head>` кожної сторінки** одразу ПІСЛЯ `<link>` на основний CSS — тобто має вищий пріоритет каскаду при однаковій специфічності. Зберігається так само в БД (themeconfig), розмір обмежений хіба полем у БД. Придатний як запасний канал для точкових овперрайдів і хотфіксів, але не для повної дизайн-системи (інлайн у кожній сторінці, без кешування, редагування через textarea адмінки).

Аналогічно існує `custom_js` (рядки 88-92) і GTM-вставки.

## 7. Висновок: найбезпечніший спосіб підключити власний astra.css

**Передумова №0 (блокер):** спочатку налагодити доставку `templates/astra/` на фронт-вузол 45.87.90.131. Поки її нема, БУДЬ-ЯКИЙ варіант підключення не працює зовні — сайт уже сьогодні тихо відкотився на 2019 для зовнішніх відвідувачів. (І це варто виправити незалежно від нашої збірки.)

Далі три реалістичні варіанти, від рекомендованого:

* **Варіант A (рекомендую): жорстке підключення в header.shared.tpl теми astra.** Оскільки astra — наша тема для адаптації, правка її шаблону легальна: замінити themeconfig-рядок на власний
  `<link rel="stylesheet" href="{$template_dir}dist/css/astra.min.css?v=<hash-вмісту>">`
  (hash вписує наш пайплайн при збірці). Плюси: повна незалежність від БД/адмінки — theme customization не зможе нічого перезаписати; власний cache-busting замість непередбачуваного `{$hb_version}`; нуль правок у БД. Мінуси: «Theme customization» адмінки для astra перестає впливати на стилі (для нас це фіча, не баг); файл custom_css-каналу зберегти як є — він і далі працюватиме поверх нашого CSS.

* **Варіант B: залишитись у штатному механізмі.** Вести адаптацію прямо в `src/scss` теми astra і дати адмінці самій компілювати `custom.min.css` (scssphp збирає тему повністю — доведено). Плюси: нічого не міняється в інфраструктурі, file_css керується як задумано. Мінуси: компілятор лише scssphp (без autoprefixer/minify), тригер збірки — руками в адмінці, версіонування — той самий непередбачуваний `?v=`.

* **Варіант C (не радив би): підміна значення file_css у БД на власний файл.** Працюватиме, але значення живе в БД у закодованій підсистемі, і перше ж збереження кастомізації в адмінці мовчки поверне `custom.min.css` — крихко і неочевидно для колег.

Спільне для всіх: **не називати власний файл `custom.min.css` / `custom_darkmode.min.css`** — ці імена належать адмінці, вона їх перезаписує (сьогоднішні файли 16:11 — живий доказ).
