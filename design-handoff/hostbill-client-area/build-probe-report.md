# Звіт: розвідка збірки SCSS теми HostBill «2019»

Дата: 2026-07-17. Джерело: тимчасова копія `/tmp/astra-build-test` (знята з `templates/2019/src`, робоча тема не змінювалась; копію після тестів видалено).

## TL;DR

**Тема збирається БЕЗ помилок усіма чотирма перевіреними компіляторами**, включно з найновішим dart-sass 1.101.0. Жодна з «небезпечних» конструкцій (`/`-ділення, `add()`, `@import`, `darken()/lighten()`) не є фатальною сьогодні — все це лише deprecation-попередження (~594 шт. у сучасному dart-sass). Міграція синтаксису не потрібна для запуску власного пайплайна зараз, але стане обов'язковою при переході на dart-sass 2.x/3.x.

## 2. Результати спроб компіляції

Команда виду `<compiler> scss/app.scss out.css`. У всіх випадках вихідний CSS **створився повністю** і містить очікувані селектори теми (перевірено `.badge-Active…`, `@font-face` Rubik, розгорнутий `add()` → `calc()`).

|---|---|---|---|---|---|---|---|
| 1 | dart-sass (standalone, GitHub releases) | **1.101.0** (актуальна) | 0 | out-dart-new.css | 566 178 Б | **немає** | ~594 deprecation (569 приховано як «repetitive») |

Розбіжності розмірів — лише у форматуванні вихідного CSS та дрібницях типу quoting; семантично збірки еквівалентні. Один помічений нюанс: libsass зберігає повний стек `font-family: "Rubik", Arial, sans-serif !important` у скороченому записі так само, як vendor-збірка, — розбіжностей у результаті це не створює.

## 3. Які конструкції «ламаються» (насправді — ні)

Розбивка попереджень dart-sass 1.101.0 за категоріями (25 показаних + 569 приховані повтори):

| Категорія | Що це в коді теми | Сьогодні | Стане помилкою |
|---|---|---|---|
| `import` | усі 22 `@import` в app.scss + внутрішні імпорти Bootstrap | warning | dart-sass **3.0** |
| `slash-div` | `$spacer / 2`, `$font-size-base / 1.4`, `$pageloader-dimension/2`, `$input-padding-y / 2` (у т.ч. всередині vendored Bootstrap `_variables.scss:306,501…`) | warning | dart-sass **2.0** |
| `color-functions` | `darken()` / `lighten()` / `fade-in()` — variables.scss, main.scss, Bootstrap | warning | майбутні версії |
| `global-builtin` | глобальні виклики builtin-функцій без `sass:` модулів (напр. `darken($link-color, 20%)` у variables.scss:114) | warning | dart-sass **3.0** |
| `if-function` | Sass-синтаксис `if(умова, a, b)` (у Bootstrap) | warning | майбутні версії |

Окремо перевірені «підозрювані»:

* **`add()`** — це НЕ builtin Sass, а функція з vendored Bootstrap (`vendor/bootstrap/_functions.scss`), яку variables.scss імпортує першим рядком саме для цього. Працює всюди; у виході коректно розгортається в `calc(1.25em + 1.34rem)` тощо.
* **`/`-ділення** — найбільш масова легасі-конструкція, але в усіх версіях 1.x dart-sass і в libsass/scssphp обчислюється правильно. Зламається лише в dart-sass 2.0+.
* **`@import` vendor-бібліотек** — усі 9 vendor-файлів + Bootstrap резолвляться без проблем у всіх компіляторах (партіали з `_`-префіксом знаходяться коректно).

Важлива відмінність від vendor-дистрибутива: в оригінальному `dist/css/app.css` є 138 входжень `-webkit-box-flex` та інших префіксів — жоден із чотирьох компіляторів їх не додає (0 входжень). Тобто **оригінальний пайплайн вендора = sass + Autoprefixer + мініфікатор**. Для піксельної паритетності зі старими браузерами до власного пайплайна треба додати autoprefixer (або свідомо відмовитись від префіксів — цільові сучасні браузери їх здебільшого не потребують).

## 4. Висновок: чим збирати і чи потрібна міграція

**Тему реально зібрати вже сьогодні будь-яким із чотирьох варіантів. Міграція синтаксису для цього НЕ потрібна.**

Рекомендації за сценаріями:

1. **Найпростіший шлях без жодних встановлень — вбудований scssphp 1.11.0** (`php /home/hostbill/public_html/vendor/bin/pscss <in> <out>`): це буквально той самий компілятор, яким HostBill сам збирає `custom.min.css` з адмінки, тому результат гарантовано збігається з штатним механізмом. Мінус: scssphp розвивається повільно і не дасть сучасних можливостей (модулі `@use`, нові функції кольору).

2. **Для власного пайплайна глибокої адаптації — dart-sass standalone** (розпакований tarball з GitHub, root не потрібен). Працює і найновіший **1.101.0** (з ~594 deprecation-warnings — шумно, але безпечно; можна глушити `--quiet-deps --silence-deprecation=import,slash-div,color-functions,global-builtin,if-function`), і **1.45.0** (майже тихий — 5 попереджень). Розумний компроміс: закріпити (pin) версію 1.x у пайплайні і не оновлюватись на 2.x без міграції.

3. **libsass (pip-пакет або sassc)** — теж працює повністю тихо, але проєкт libsass офіційно deprecated; для нового пайплайна брати не варто.

**Коли міграція таки знадобиться:** при переході на dart-sass 2.x (зламається `/`-ділення) і 3.x (зникнуть `@import` та глобальні builtin-функції). Обсяг: `sass-migrator divide` + `sass-migrator module` покривають більшість автоматично, але доведеться мігрувати й vendored Bootstrap 4 (він сам — головне джерело легасі-синтаксису: сотні slash-div у `_variables.scss` та міксинах), тож реалістична стратегія на далеку перспективу — або тримати pinned dart-sass 1.x, або при міграції одночасно оновлювати Bootstrap до 5.x, що вже виходить за межі «рескіна».

**Ще один практичний висновок:** оскільки збирається навіть штатним scssphp, глибоку адаптацію можна вести прямо в SCSS-сирцях нової теми (клон `astra`), не ламаючи вбудований механізм кастомізації HostBill — адмінка й далі зможе перекомпільовувати тему зі своїми змінними.
