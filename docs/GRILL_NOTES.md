## Зафіксовані рішення

1. **Архітектурний патерн ізоляції HostBill — переноситься.** HostBill —
   деталь реалізації, ізольована за портами/адаптером (`billing-adapter`);
   жоден інший модуль не викликає HostBill API напряму; scoped API-ключі
   (окремі ключі під кожен сценарій, не "ключ на все"); webhook перевіряється
   за підписом і обробляється ідемпотентно (за event ID); internal `userId`
   ≠ HostBill `client ID` з першого комміту; дані картки ніколи не проходять
   і не зберігаються на нашій стороні (hosted payment page).
   Конкретний стек зі старого проєкту (NestJS/Next.js/Payload/Redis+BullMQ) —
   **не** переноситься автоматично, обговорено окремо (див. п.4).

2. **Скоуп зараз:** тільки публічна маркетингова частина + self-checkout.
   Особистий кабінет (client-portal) буде інтегровано **пізніше** — користувач
   додасть свої шаблони окремо, коли вони будуть готові. Зараз після оплати —
   редірект у вже існуючий HostBill client area (`cp.astra.in.ua`), без змін.

3. **Порядок реалізації сторінок** (кожна — окрема spec-kit фіча, не одна
   велика):
   1. Фундамент — інтеграція дизайн-системи/компонентів у обраний стек,
      налаштування CMS-схеми і мовної архітектури.
   2. Homepage.
   3. VPS page (додає `PricingTable` із фільтрами: локація/CPU/IPv4-IPv6/оплата).
   4. Checkout (`vps_checkout`) — найскладніша, multi-step wizard.
   5. Simple template (генерична сторінка) — можна й раніше в черзі, як дешевий
      тест лейауту.

4. **Технологічний стек — Варіант B (спрощений моноліт).** Один Next.js
   застосунок (App Router) з **вбудованим** Payload CMS (нативна інтеграція
   в той самий Next.js-процес, без окремого CMS-сервісу). Білінг-адаптер —
   ізольований серверний модуль (route handlers/server actions) усередині
   того самого застосунку. PostgreSQL для даних. Redis/BullMQ — **не одразу**;
   вебхуки HostBill обробляються синхронно, ідемпотентно (перевірка за
   event ID в Postgres), черга додається пізніше лише якщо навантаження
   виправдає. Причина вибору: команда без штатних програмістів (адміни/DevOps
   + AI-агенти) — пріоритет простоти підтримки над архітектурною витонченістю.

5. **Провіжинінг серверів — поза скоупом.** Наша система відповідає лише за
   шлях клієнта до оплати: конфігурація → чекаут → створення Order/Invoice в
   HostBill → відображення статусу після вебхука (оплачено/помилка). Що
   відбувається з фізичним/віртуальним сервером далі — залишається на стороні
   HostBill/адмінів, як і зараз.

6. **Автентифікація/сесії — не будуються зараз.** Немає власного логіну/сесій,
   доки немає кабінету. Крок 2 чекауту — легкий pass-through до HostBill:
   - onBlur перевірка email через HostBill (`GetClients`-подібний метод,
     ключ `portal-readonly`) — лише щоб перемкнути UI між "реєстрація"/"логін",
     без створення сесії;
   - новий клієнт: дані форми живуть тільки як чернetка чекауту (short-lived
     cookie/draft), пишуться в HostBill (`AddClient`) тільки на кроці 3;
   - існуючий клієнт: email+пароль передаються напряму в HostBill
     (`ValidateLogin`-подібний метод) в тому самому запиті, пароль ніколи не
     зберігається і не логується на нашій стороні;
   - `Client`/`Order`/`Invoice` створюються в HostBill тільки на кроці 3
     ("Оформити замовлення"), не раніше.
   Але **таблиця мапінгу `internal_user_id ↔ hostbill_client_id`** заводиться
   з першого комміту (навіть без активної автентифікації) — щоб усі майбутні
   внутрішні дані посилались на internal_user_id, а не напряму на HostBill ID,
   і щоб кабінет пізніше міг "приєднати" реальний логін до вже існуючого
   запису без міграції даних.

7. **Мовна архітектура:** UA/EN/PL, path-based роутинг (UA — дефолт без
   префіксу в URL, `/en/`, `/pl/`). Повна i18n-архітектура (роутинг, схема CMS
   з `localized`-полями) закладається одразу під усі 3 мови, але **реальний
   переклад зараз — тільки UA**; EN/PL — fallback/заглушка до появи перекладів.

8. **Валюта:** перемикач UAH/USD/EUR потрібен (підтверджено з `home.txt`,
   шапка сайту explicitно містить "Вибір валюти (UAH / USD / EUR)").

9. **Джерело цін/специфікацій для PricingTable — HostBill, не CMS.** Ціни й
   характеристики (CPU/RAM/диск) тягнуться напряму з HostBill через окремий
   порт billing-adapter'а (напр. `ProductCatalogProvider`), не дублюються
   вручну в CMS. CMS відповідає лише за маркетинговий текстовий шар навколо
   картки тарифу (бейджі, описи, порядок). Мокап у хендофі — лише демонстрація
   вигляду, не джерело реальних даних.

10. **"Колокація" та "Виділений сервер" — заглушки на `simple`-шаблоні
    зараз.** Обидва є в меню й мають тизер-блоки на головній, але власних
    дизайн-мокапів для них ще немає. Тимчасово ведуть на generic
    `simple`-шаблон із базовим текстом; повноцінні посадкові сторінки —
    окрема майбутня фіча, коли з'явиться дизайн-хендоф.

11. **Моделювання контенту в Payload — `blocks`-масив з першого дня, тільки
    для контентних (маркетингових) сторінок.** ФІНАЛЬНЕ рішення (переглянуто
    після додаткового обговорення):
    - Кожна **контентна сторінка** (homepage, VPS/Dedicated/Colocation landing,
      `simple`-генерик) моделюється як Payload-колекція з полем `blocks`
      (реальний масив, не fixed group) — редактор сам додає/прибирає/
      переставляє блоки на сторінці без участі агента/розробника.
    - **Список дозволених типів блоків — курований і обмежений тим, що
      реально є в дизайн-мокапах** (`.dc.html` файли в `design-handoff/project/
      templates/`). Не вигадувати нові типи блоків "про запас" — тільки ті,
      що вже спроєктовані (Hero, TrustBar "Нам довіряють", FeatureGrid/
      переваги, UseCase, FAQ, ConsultationCTA "Потрібна допомога з вибором?",
      PricingTable-варіанти, тощо — точний перелік по кожному шаблону
      уточнюється під час `/speckit-specify` для відповідної сторінки, читаючи
      сам `.dc.html`).
    - Кожен блок визначається **один раз** як переюзаний field-конфіг +
      React-компонент; список дозволених блоків можна обмежувати **окремо
      для кожної сторінки/колекції** (напр. `VpsPricingTableBlock` доступний
      тільки в межах VPS-сторінки, не пропонується на Homepage).
    - **Дата-блоки** (напр. `VpsPricingTableBlock`, `DedicatedPricingTableBlock`,
      `ColocationOfferCardsBlock`) — особливий підтип: CMS-схема тримає лише
      маркетингову обгортку (заголовок, slug групи товарів у HostBill,
      параметри відображення), а самі ціни/специфікації підтягуються "живими"
      з HostBill через `billing-adapter`/`ProductCatalogProvider` (див. п.9)
      під час рендеру, з коротким кешем.
    - **Чекаут і кабінет (client-portal) — НЕ на блоках.** Це транзакційні/
      застосункові сторінки з фіксованою логікою (wizard зі станом, таблиці
      даних із бекенду) — код, не CMS-композиція. CMS там може постачати хіба
      що окремі i18n-мікротексти (лейбли, повідомлення про помилки) або
      точково визначені вставки (напр. один банер), але не вільний
      `blocks`-масив на всю сторінку — це навмисно, щоб редактор не міг
      випадково зламати платіжний флоу перетягуванням блоків.

12. **Деплой/хостинг.** Власний сервер Astra (self-hosted), не зовнішній
    PaaS. Локальна розробка: Next.js-застосунок запускається "голим" (без
    Docker) для швидкої ітерації й нативного дебагу; Postgres — у Docker
    локально. На проді Postgres **теж у Docker**, той самий образ/версія
    (`postgres:16` і те саме число скрізь) — це знімає питання розбіжності
    версій між середовищами. Сам застосунок на проді запускається процес-
    менеджером (pm2 або systemd-юніт) за реверс-проксі (Nginx/Caddy — SSL,
    роутинг); чи буде застосунок теж у Docker на проді — вирішується пізніше,
    без обов'язкового збігу з локальним підходом. **Staging-середовище — не
    робимо**, свідомо спрощуємо (тільки local + production). Секрети на
    проді — через deploy-скрипт (не ручне розкладання `.env`), деталі
    скрипта — за реалізації. Міграції Payload/Postgres — явний, автоматичний
    крок деплою **перед** перезапуском процесу (git pull → міграції → білд →
    рестарт), ніколи не вручну "коли згадаю".
    **CI/CD — GitHub Actions.** Агент пише сам workflow
    (`.github/workflows/deploy.yml`) і deploy-скрипт (це звичайний код,
    безпечно робити відразу) під час реалізації фази "Фундамент". SSH-доступ
    до реального сервера (хост/користувач/приватний ключ) — це секрет, який
    користувач додає сам через GitHub Secrets, коли сервер буде готовий;
    агент лише посилається на нього за назвою в workflow, не отримує значення
    в чаті.

13. **Інженерні практики зі старого проєкту — переносяться.** TypeScript
    strict mode, `eslint-plugin-boundaries` (технічна заборона крос-модульних
    імпортів — ізоляція billing-adapter навіть у межах моноліту Варіанту B),
    Semgrep SAST, підвищене тестове покриття для `billing-adapter` — усе це
    не залежить від дизайну/бренду й особливо цінне без штатних програмістів
    (лінтер/SAST ловлять те, що нема кому зловити код-рев'ю).
